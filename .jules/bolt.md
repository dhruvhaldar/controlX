
## 2024-03-24 - Vectorizing control.frequency_response() over python loops
**Learning:** In the `python-control` library, evaluating system frequency response inside a python loop using `ct.evalfr(sys, w*1j)` is extremely slow. Using the vectorized `sys.frequency_response(omega).complex` returns a multidimensional array (e.g. `(n_outputs, n_inputs, n_frequencies)` for MIMO) containing all frequencies at once. By transposing this to `(n_frequencies, n_outputs, n_inputs)` and using `np.linalg.svd` with `compute_uv=False`, we can perform a batched SVD computation across all frequencies simultaneously. This achieved ~6x speedup for MIMO systems and ~90x speedup for SISO systems.
**Action:** When working with frequency sweeps in `python-control`, always prefer the vectorized `sys.frequency_response(omega)` API over manual loops. Additionally, remember to correctly handle the resulting dimensional array shape (e.g. using `np.transpose` for batched `np.linalg.svd`) to maximize vectorized performance.

## 2024-05-18 - Parameterizing CVXPY problems for MPC
**Learning:** In MPC implementations using `cvxpy`, rebuilding the optimization problem (`cp.Variable`, `cp.Problem`, constraints, and cost function) at every time step is extremely slow due to the overhead of parsing and compiling the problem. By defining the problem once in the `__init__` method and using `cp.Parameter` for the initial state `x0`, we can avoid recompilation. Setting the parameter value and solving with `warm_start=True` provides over 10x speedup (e.g. 2.8s -> 0.2s for 50 steps of a simple 2x2 MIMO system).
**Action:** When implementing iterative Model Predictive Control (MPC) with `cvxpy`, always parameterize the problem using `cp.Parameter` for variables that change between iterations (like the initial state). Set up the problem once during initialization and only update parameter values before calling `solve(warm_start=True)` in the control loop.
